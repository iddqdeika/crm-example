# Research: Adtech Landing & Profile Auth

## Technology & Architecture Decisions

### Backend stack (FastAPI + PostgreSQL)

- **Decision**: Use Python 3.12 with FastAPI, SQLAlchemy, and PostgreSQL for the backend.
- **Rationale**: FastAPI provides high-performance async APIs, strong typing, and automatic
  request validation, which aligns well with testability and TDD. PostgreSQL is a robust,
  ACID-compliant database with good JSON support and strong ecosystem support for auth and
  profile-style data.
- **Alternatives considered**:
  - Node.js (Express/NestJS) + PostgreSQL: viable but less aligned with the existing
    Python-focused constitution examples and would duplicate infrastructure for other
    Python-based tools.
  - Django: strong for auth and admin but heavier-weight for a microservice-oriented,
    API-first approach.

### Frontend stack (React + TypeScript)

- **Decision**: Use React with TypeScript for the web frontend.
- **Rationale**: React + TypeScript is a widely adopted standard for SPAs in 2026 and is well
  supported by modern tooling and testing libraries. TypeScript improves correctness for
  auth/profile flows and aligns with the constitution’s emphasis on testability.
- **Alternatives considered**:
  - Vue or Svelte: also capable, but React has broader ecosystem support and more examples for
    adtech-style landing experiences.
  - Pure server-rendered HTML: simpler but less flexible for interactive profile management.

### Authentication model

- **Decision**: Email + password authentication with session-based auth using HTTP-only cookies.
- **Rationale**: Session cookies are a well-understood, secure default for web apps, reducing
  the risk of token leakage compared to manually managed JWTs in local storage. They work well
  with FastAPI and modern reverse proxies and are straightforward to test.
- **Alternatives considered**:
  - Pure JWT-based stateless auth: more complex revocation and session management; better
    suited for many heterogeneous clients than this initial web-focused feature.
  - Third-party auth providers (OAuth/Social login): useful later, but out of scope for the
    initial profile + password flows.

### Database migrations

- **Decision**: Use Alembic for schema migrations, with a dedicated `backend/migrations/`
  directory and scripts to run migrations from Dockerized environments.
- **Rationale**: Alembic is the de facto standard for SQLAlchemy-based projects and integrates
  well with FastAPI. It supports autogenerated migrations while still allowing manual control.
- **Alternatives considered**:
  - Custom SQL migration scripts: less tooling support and more manual bookkeeping.
  - External migration tools: add complexity without clear benefit for a single backend
    service.

### Dockerization & configuration

- **Decision**: Provide separate Dockerfiles for backend and frontend, plus Docker Compose
  definitions (`docker/docker-compose.dev.yml`, `docker/docker-compose.prod.yml`), with all
  configuration injected via environment variables.
- **Rationale**: Separate images align with microservice-friendly design and allow independent
  scaling and deployment. Env-based configuration follows 12-factor app practices and matches
  the constitution’s requirement that deployments be expressed as code with externalized
  config.
- **Alternatives considered**:
  - Single container for backend + frontend: simpler but less flexible and less aligned with
    microservices by design.
  - Hard-coded config files: conflict with Docker-first and environment-based configuration
    requirements.

### Startup and migration scripts

- **Decision**: Implement `backend/scripts/start.sh` to run migrations then start the app, and
  `backend/scripts/migrate.sh` to run Alembic upgrade/downgrade commands explicitly.
- **Rationale**: Explicit scripts make it easy to run migrations in CI/CD, local dev, and
  production containers, and they give a single entrypoint for Docker CMD/ENTRYPOINT that
  respects the constitution’s Docker-first requirements.
- **Alternatives considered**:
  - Running migrations implicitly inside application startup: hides operational behavior and
    makes failures harder to detect and control.

### Cursor rules for main technologies

- **Decision**: Create Cursor rules for FastAPI/Python, React/TypeScript, and Docker/env-based
  configuration under `.cursor/rules/`, with focused guidance per technology.
- **Rationale**: These rules will help keep implementation consistent with the chosen stack
  and with the constitution (TDD, Docker-first, microservices), and they provide persistent
  context for future agents working on the project.
- **Alternatives considered**:
  - Single monolithic rule file for all technologies: harder to maintain and less targeted.
