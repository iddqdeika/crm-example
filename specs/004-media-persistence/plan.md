# Implementation Plan: Media Persistence (Avatar Storage)

**Branch**: `004-media-persistence` | **Date**: 2026-02-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-media-persistence/spec.md`

## Summary

Replace ephemeral local-filesystem avatar storage with a durable, standalone **MinIO** object storage service (S3-compatible). Introduce a thin `StorageClient` abstraction in the backend so that `avatar_service.py` is decoupled from the storage technology. MinIO runs as a dedicated Docker Compose service with a named volume. Avatar URLs are changed from backend-proxied paths to **presigned URLs** generated by MinIO, served directly to the browser. No frontend code changes are needed.

## Technical Context

**Language/Version**: Python 3.12 (backend), TypeScript / React 18 (frontend вЂ” no changes)  
**Primary Dependencies**: FastAPI, SQLAlchemy async, `boto3` (S3 client), MinIO Docker image  
**Storage**: PostgreSQL (metadata), MinIO/S3 (binary objects)  
**Testing**: pytest + pytest-asyncio (backend); Vitest (frontend вЂ” no new tests needed)  
**Target Platform**: Linux container (Docker Compose, dev and production)  
**Project Type**: Web service (FastAPI microservice + MinIO standalone service)  
**Performance Goals**: Avatar upload/retrieval <500ms p95 in local dev  
**Constraints**: Files up to 5 MB; only authenticated users may retrieve avatars (via presigned URL); MinIO must persist data across container restarts  
**Scale/Scope**: Single-tenant dev instance; architecture supports multi-tenant via bucket/prefix namespacing

## Constitution Check

### вњ… Testability First

- `StorageClient` is a Protocol в†’ fully injectable and mockable without a real MinIO instance.
- `FakeStorageClient` (in-memory) is provided in `backend/tests/fakes/` for unit and contract tests.
- All new behavior is verifiable without Docker networking.

### вњ… Test-Driven Development

- Unit tests for `avatar_service.py` (upload, remove, presigned URL) are written **before** modifying `avatar_service.py`.
- Contract tests for `POST /me/avatar`, `DELETE /me/avatar` are written **before** modifying `api/profile.py`.
- Each task in `tasks.md` has a corresponding test task that precedes it.

### вњ… Red-Green-Refactor

- All new tests start in the Red state (failing) and turn Green only after the implementation task.
- The `FakeStorageClient` enables immediate local Redв†’Green loops without Docker.

### вњ… Microservices by Design

- MinIO is a **standalone service** in Docker Compose. The backend communicates with it only through the S3 HTTP API.
- The backend owns `Avatar` metadata in PostgreSQL; MinIO owns the binary objects. No shared state.
- The `StorageClient` abstraction is the versioned, testable contract between the backend service and the storage service.

### вњ… Docker-First Delivery

- MinIO is added to `docker/docker-compose.dev.yml` with a named volume (`minio_data`).
- A `minio-init` service runs `mc` to create the bucket on first start.
- The same compose file is used for local dev and CI. No new Dockerfiles needed for the backend.
